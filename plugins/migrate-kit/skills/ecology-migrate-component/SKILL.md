Execute the migration of **one** reusable primitive from an `ecology-audit` plan to its Ecology target. The audit said *what* and *in what order*; this skill does the *swap* for a single component: reimplement it in Angular faithful to the Ecology spec, reconcile its public API, update its consumers, and verify. Operate on one component (or one slice) per run.

## Consumption model (read first — it determines HOW you swap)

The hub's interactive components are **Lit web components** — a different technology from Angular, and **not published for Angular consumption**. An Angular app **never imports them** (there is no future "install the package" step); it **reimplements the esa-* contract as its own permanent Angular component library**, with **Beacon's `ui-*` as the Angular reference implementation** and the hub's `.astro` + tokens as the structure/token contract. So:

- **Presentational target (`.astro` in the hub)** → **reimplement as an Angular lego.** The hub `.astro` is the **token + structure contract**; **Beacon's `ui-*` is the Angular reference** (the `.astro` is literally a "faithful translation" of it). Build a token-driven Angular component; import nothing from Ecology.
- **Interactive target (Lit `.ts` in the hub)** → **reimplement as an Angular lego: Beacon's `ui-*` is the API/structure reference, but you MUST read the hub Lit component's `static styles` — they are the VISUAL contract (the spoke renders *that* component, not Beacon's).** Do **not** assume Beacon's look matches the hub: **Beacon is a different app** and its `ui-*` can carry its own styling that diverges from the hub (e.g. Beacon's `ui-toast` is a soft/light tinted-with-left-border toast; the hub's `esa-snackbar-item` is a solid-DARK toast — opposite looks). When they differ, **the hub wins on visual, Beacon on API/variants.** The Lit WC is never an Angular import — but its `static styles` ARE the look to mirror (translate them to the app's tokens, with the hub's literal values as fallbacks).
- **COEXIST (ag-Grid surface)** → no swap; theme via `--grid-*` tokens only.

These Angular legos are **permanent** — the reimplementation *is* the deliverable, not a placeholder for a later import.

In every case the esa-* lives as a **reusable lego the app's components compose** — a standalone Angular component, not styling pasted into each consumer (see Phase 2). The component axis *builds the lego library* and migrates consumers onto it.

## Arguments

`$ARGUMENTS`:
- **`--component <path|name>`** — the primitive to migrate (required). Resolve against the audit's primitive list.
- **`--app <path>`** / **`--ecology <path>`** / **`--beacon <path>`** — app, Ecology hub, and Beacon repo roots. Defaults: current repo; `../ecology`; `../Beacon`.
- **`--target <esa-*>`** — the Ecology target; default from `component-audit.md`.
- **`--dry-run`** — produce the reimplementation + consumer-change plan, modify nothing.

## Workflow

### Phase 0 — Resolve inputs
**Component-specific reference.** If a `components/<target>.md` exists next to this skill (e.g. `components/button.md`), **read it first** — it carries the component-specific target API, the legacy-class→variant mapping, glyph maps, what to leave alone, the legacy stylesheet to retire, and the batch areas. This `SKILL.md` stays general; per-component specifics live there (and a batch agent should be pointed at the `.md`, not handed a re-pasted recipe).

From the audit artifacts (`docs/ecology-migration/`): the component's **classification + target** and its slice. Then locate the **three references**: (1) **the spoke's design-system page** — `<spoke>/src/pages/design-system/components/<target>.astro` — the **project's ground-truth** for which variants/sizes/props the design actually uses, its summary/usage notes, and (via the theme it inherits) the target token *values*; (2) the **Beacon `ui-*` reference** (the analogue under `<beacon>/Beacon.Web/src/app/shared/ui/components/`) — the proven Angular API + structure; (3) the **hub `.astro`/`.ts` spec** — the token + structure contract. **Always open the spoke page** (even when the run didn't re-pass `--spoke` but the audit was spoke-driven) — *skipping it is the easy miss*; it's how you confirm the lego matches the design the project actually adopted. **Also check the spoke's handoff** (`<spoke>/src/data/handoff/<screen>.mjs`) for a section matching this component (a "Page header" / "Filters" / … section): its `decisions`/`gotchas`/`acceptance` are the most specific, *screen-grounded* design spec and **override the generic hub `.astro`** where they differ — a `gotcha` is a literal "don't do X" to honor (e.g. the page-header handoff pinned the title to **primary blue at weight 500, the same blue as the icon**, with an explicit gotcha against the hub's default near-black/semibold). **Don't stop at the prose `decisions`** — read the generated `public/handoff/<screen>/claude/<section>.md` (its **Styles + Markup blocks carry the EXACT CSS**: line-heights, gaps, letter-spacing, fonts) and cross-check the real spoke component; the prose skims, the CSS is the spec. **Read the Markup block's component classes/variants too** (`class="esa-icon esa-icon--md"` ⇒ size `md`), and **when a value still looks off, MEASURE the live published prototype** (`getBoundingClientRect`/computed styles) — a Styles rule in the doc can be *overridden* in practice and never apply. (The page-header true-up first missed the font/line-heights/gap by reading only the prose decisions; then it over-sized the icon to 28px by trusting the Styles block's `svg { width: 28px }` — a dead rule, since the markup said `esa-icon--md` and the live render is 20px because esa-icon's own md sizing wins. Match the markup + the measured live render, not an overridable CSS rule.) **If the spoke has *no* page for the target** — a NET-NEW hub component promoted from other spokes' demand (e.g. `esa-collapsible`) — that's expected, not a blocker: build from Beacon + the hub `.astro` and **flag the absent spoke page** in the report (no project-specific variant guidance; the spoke's theme values still flow through `--color-*`). Confirm the app's token bridge exists (the Ecology semantic aliases in the theme file). If the target is COEXIST, stop and emit token-theming guidance instead of a swap.

**Target tie-break.** If the audit lists more than one candidate target and `--target` wasn't passed, **prefer the one that has a Beacon `ui-*` reference** — you must be able to translate against a real Angular implementation (per the consumption model). If none has a Beacon analogue, pick the closest by role and **state in the report that you worked from the `.astro` spec alone** (higher risk).

### Phase 1 — Capture the current contract (before touching anything)
Record the component's **selector, `@Input()`/`@Output()` (and signal `input()`/`output()`), content projection, and behavior**, and enumerate its **consumers** (every template/TS that uses its selector or imports its class — this is the blast-radius you must keep working). **Exclude generated artifacts** when enumerating consumers — `shared/generated/**`, compodoc `documentation.json`, and build output produce false hits that inflate the blast-radius. This contract is what you preserve or deliberately migrate. For a native-element absorption (`button`/`input`/…) the consumer set is **every** raw element — inventory it **exhaustively from the grep, file by file**, not by hand-grouped feature areas; stray pages (help, home, login, a style-guide) and shared toolbar components are easy to miss. **Also inventory *hand-rolled inline duplicates* of the same primitive — not just the bespoke component's selector.** The same pattern often exists as raw markup inlined in a page (a `<dl class="detail-list">` reimplementing the `key-value-pair` component; a `<div class="card">` reimplementing a card) that a selector / class-import grep never sees. Grep for the primitive's **structural / CSS-class signature** (the BEM block, `.detail-list`, `.stat`, the element + its telltale classes) across templates **and** scss, and fold those consumers + their bespoke stylesheets into the same migration — otherwise the lego ships while copies of it linger (and a reviewer finds them before you do). When a consumer's inline duplicate carries a look the lego lacks (e.g. a boxed value), add it as a **mode/variant on the lego** and compose it — don't leave the duplicate, and don't bake the look into the page.

### Phase 2 — Build the Ecology-faithful Angular implementation
Mirror the reference; do **not** invent a different API or visual. From the Beacon `ui-*` + the hub `.astro`:
- **Structure:** standalone component, `OnPush`, host-class variant pattern (`class="x x--{variant}"`), content projection via `<ng-content>`, inputs per the app's convention (signal `input()` or `@Input()` — match the app). For a **nav/link primitive** (breadcrumbs, link-column, nav-dropdown, icon-link), the hub `.astro` uses a static `href`; the Angular lego uses **`routerLink`** (client-side nav) — adapt the binding, keep the structure. `OnPush` is the default, **but keep *default* change detection for a lego that projects consumer-supplied `TemplateRef`/content bound to the consumer's changing data** (e.g. a page-header with `templateRight`/`templateBottomRight` slots) — `OnPush` can stale the `ngTemplateOutlet`/projected content, since it only re-checks on an input change or an internal event.
- **Styling:** SCSS using **component-tier CSS-custom-property hooks that default to the app's Ecology-aliased semantic tokens** (e.g. `--badge-bg: var(--badge-bg-success, var(--color-success))`). **Translate** Beacon Sass vars / hub token names → the app's `var(--…)` names. **Flag** any token rung the app lacks (e.g. `--spacing-150`) rather than hardcoding — add it to the theme's alias bridge or note it for the token gap.
- **No hardcoded brand values** — read tokens so the design spoke's final values flow through.

**Mirror-vs-match split (they don't conflict).** *Mirror the reference* for **structure, variants, and visual** — the things that make the swap faithful. *Follow the app* for **API-surface style** — `@Input()` vs signal `input()`, file layout, naming. Faithfulness is about behavior and appearance, not decorator syntax; so an app that is uniformly `@Input()`-based stays `@Input()` even if the Beacon reference uses `input()`.

**Build the esa-* primitive as a standalone reusable lego — side by side — then compose it; don't reimplement in place.** The esa-* targets form a reusable *component library*. Create each as its own Angular component in a dedicated legos dir (e.g. `<app>/src/app/shared/components/esa/<name>/`), selector **`esa-<name>`** (mirrors the hub's custom-element tag + Beacon's `ui-*`), class `Esa<Name>Component` — built *alongside* the app's current usages, not by overwriting an app component. Then (Phase 4) migrate consumers to **compose** the lego and retire the bespoke duplicates.
- **A true-1:1 primitive may *become* the lego in place.** When an app component already *is* the single shared primitive (the app's one shared `button` → `esa-button`, its one `icon` → `esa-icon`), it can become the lego where it lives — same role, it already was the shared thing (keep or relocate its path).
- **A domain component that merely *renders* a primitive must COMPOSE the lego.** A component that uses a primitive but isn't itself the reusable lego (a fullscreen toggle that renders an icon-button; a domain card built from a button + badge) composes the lego — it must **never bake the lego's styling into itself**. That baking is the bug this rule prevents: it duplicates the lego (the shell, the grid toolbar, and the next screen each grow their own copy) and never builds the library.
- **Build foundational legos first.** Legos compose each other — `esa-icon-button` renders an `esa-icon`, which needs the Lucide icon registry; `esa-button` renders an `esa-icon`. Build the leaf legos (`esa-icon` + the icon registry) before the composites, and have composites import the lego, not re-inline its markup.

**Spoke-vs-Beacon-vs-hub precedence.** Three references, three jobs — they rarely conflict, and when they do:
- **Spoke design-system page = ground-truth for *what the design uses*** — the variants/sizes the project actually adopted + (via its theme) the token *values*. Reconcile the lego to it. But its API table is usually the **common surface** (the inherited hub component's headline props), **not a narrower contract** — a prop Beacon has that the spoke page omits is a **legitimate superset**, not a divergence; keep it and note it (e.g. a button's `dashed`/`soft` appearance, an icon-button's `color`/`active`). Don't trim a Beacon-sourced prop just because the spoke doc didn't enumerate it.
- **Beacon `ui-*` = the proven Angular API / variants / structure** — mirror it; it wins on API breadth + behavior.
- **Hub `.astro` *or* Lit `.ts` `static styles` = the token + structure contract — and the VISUAL contract whenever Beacon's look differs.** For an interactive target there's usually **no `.astro`**; the **hub Lit component's `static styles` are the visual truth** (the spoke renders that component) — open and read them, don't infer the look from Beacon. Beacon is a different app: treat its `ui-*` visual as a *hint*, not gospel. When Beacon and the hub diverge (Beacon's badge soft vs hub solid; Beacon's toast light vs hub solid-dark), **match the hub and say so.** **"Match the spoke" = the hub component's structure + its token *names*, resolved against the spoke's theme — not the hub's generic rendering.** Read the *same tokens the hub component reads*: tokens the spoke **overrides** (brand `--color-primary`/`-secondary`, fonts, radius) then flow to the spoke's values automatically, while tokens the spoke **inherits** (status/neutral palette — common, since a spoke usually overrides only brand/font/radius) resolve to the hub default. So for a component whose variants read the inherited *status* palette, the hub's per-variant colors ARE the spoke's resolved colors — mirror the hub's token chains. The literal fallback is only a safety net for a token the app's theme doesn't ship; set it to **what the spoke resolves that token to** (the hub default for an inherited token), **never a hardcoded brand color**.
Say which reference drove each decision in the report.

### Phase 3 — Reconcile the API
Compare the old contract (Phase 1) to the target's variant model. For each gap pick one, and **say which in the report**:
- **Preserve** the old selector + inputs (lowest blast) and map old values to the new model internally (e.g. legacy arbitrary `color` → nearest semantic `variant`; keep a deprecated `@Input` with a console-deprecation note).
- **Migrate** the API (cleaner) and update every consumer in Phase 4.
- **Drop** an input that has no representable mapping in the target's model **and** that no consumer relies on for an off-palette/custom value (e.g. an arbitrary `color` hex where every consumer's value maps cleanly to a semantic variant). Removing it is then the *point* of the swap — say so and list the value→variant mapping you applied.
Default to **preserve** for low-blast/leaf primitives unless the old API is the thing being removed.

### Phase 4 — Migrate consumers to the lego, then retire the bespoke version
Swap each consumer from Phase 1 to **compose the new `esa-*` lego** — replace the old tag/markup with `<esa-name …>`, or, for a domain wrapper, have that wrapper render the lego internally. (If the migrated component was a true-1:1 primitive that *became* the lego in place, this is the Phase 3 API-reconcile update instead — a no-op when the API was preserved; say so.) Never leave a consumer half-migrated. **Retire the bespoke original:** once nothing references the old component, **delete** it (files + dir); if a consumer can't move yet, leave the old as a thin **deprecated shim that forwards to the lego** — say which you did. End with **one** implementation of the primitive (the lego), never a copy per consumer.

**Migrating native-element consumers — generalizable rules** (the component-specific specifics — the legacy-class→variant map, the lego's icon API, glyph names, what to leave alone — live in `components/<name>.md`):
- **Drop only the tokens you're migrating; keep the rest.** When you strip a legacy variant class, drop just those tokens and preserve any other utility/layout class on the same element.
- **Legacy state hooks → native inputs.** State expressed via a legacy-stylesheet class (a cosmetic disabled/loading class whose look lived only in the old global stylesheet) maps to the lego's **native input/property**, not a now-dead class — often restoring behavior the cosmetic class never had. An existing native binding is kept as-is.
- **An explicit variant class wins over a default heuristic.** Map a consumer by the variant its markup actually carries; a default mapping (e.g. a cancel convention) applies only when the markup gives no variant.
- **Icons: input for leading, projection for the rest.** A lego with an icon input takes one *leading* glyph via the input; *trailing* / both-side / extra glyphs are projected into content. Map **every** legacy glyph source — an icon font (`<i class="…">`), the app's own icon component, or a literal text char in the label — to the lego's icon system, and **keep the app's icon component imported** if a non-migrated glyph remains elsewhere in the template. When you change a component's icon **input type** (an app `IconInterface` → a Lucide-name string), also catch **dynamic `[icon]` bindings + icon-name lookup maps** (`[icon]="someVar"`, an `activeTabIcon: {...}` map) — a literal `icon="…"` grep misses them, and the type change breaks them *silently* (the value still type-checks as a string but isn't a registry key).
- **Import-only edits can trip a repo guardrail on pre-existing code.** Adding just an `imports` entry can fire an rxjs/lint hook over *unrelated, pre-existing* code in the file (`.subscribe()`/side-effects, or plain DTO/container classes with public array fields). Not a regression — don't "fix" out-of-scope code; note it and move on.

### Phase 5 — Verify
**First, run the app's documented formatter on every file you created or modified** (e.g. `npx prettier --write <files>`, per the app's `.prettierrc`/conventions) — the build *and* the verify page pass regardless of formatting, so unformatted files ship silently and surface only in review. Then run the app's **documented** production build **from the Angular project root** and confirm success — e.g. `npm --prefix <web-dir> run build`, or `cd <web-dir> && ng build`. Do **not** assume `npx ng build` works from the repo root: when the Angular project lives in a subdirectory it won't find `angular.json`. For a multi-file native-element absorption, **a passing build is not completeness** — un-migrated legacy elements still compile fine. **Do a final marker sweep** to catch consumers the area-batches missed — grep the legacy class **by variant name / with word boundaries** (e.g. `grep -rnE '\bbtn-(primary|secondary|danger|sm|lg|icon)\b'`), **NOT** `class="btn` (that only matches `.btn` as the *first* class and misses `class="feature__link btn btn-primary"`). Sweep **both templates and component `.scss`** (orphan style rules linger after a template migrates). The migration isn't done until that sweep is clean, modulo the documented config-driven / other-slice residue. **Then generate the verify page — the closing deliverable of the migration, not an optional handoff.** Run **`ecology-verify --component <name>`** (mode A isolation) to produce the self-contained before/after harness HTML + screenshots + the regenerated navigable index under `.playwright-mcp/ecology-verify/`. `ecology-verify` builds its **own** harness (the theme `:root` tokens + scoped before/after component CSS + representative instances) — it does **not** render the app's style-guide, so the lego does **not** need to be wired into any app page to be verified (don't add it to a style-guide/gallery for verification's sake). "Build passed" is necessary, not sufficient — **completion means the review page exists** for a human (or the conductor's gate) to eyeball. Note: a reference-faithful reimplementation may intentionally change a *legacy* visual (e.g. the old component force-uppercased its text and the reference doesn't); that's expected — appearance deltas are adjudicated in `ecology-verify`, not reverted here.

### Phase 6 — Report
State: the component, target, consumption mode (reimplement/vendor/coexist), the API decision (preserve vs migrate) + any deprecations, consumers touched, tokens read + any missing rungs flagged, libs retired, and the **`ecology-verify` before/after page path + its PASS/FLAG verdict** (generated as Phase 5's closing step) — tell the user to open the index to review. End with the reminder that this swap is skill-produced — to change it, reset → refine this skill → re-run.

## Notes
- **Build the library, compose — don't scatter.** Each esa-* is one reusable lego that domain components compose; reimplement-in-place only for a true 1:1 shared primitive. Never duplicate a lego's styling into a consumer — if two components would carry the same esa-* markup/CSS, that markup belongs in a lego they both import.
- **A primitive that styles a native element absorbs ALL its raw usages.** When the esa-* is the design-system version of a native HTML element (`button`→`esa-button`, `input`→`esa-text-field`, `select`→`esa-select`), its consumer set is **every raw `<button>`/`<input>`/… in the app**, not just an existing wrapper (`fresca-button`) — annotate each native element with the directive (`<button esa-button color="…">`, keeping native `type`/form/a11y semantics). Classify each raw usage: standard → the element directive; **icon-only → `esa-icon-button`**; leave bespoke chrome triggers (menu/toggle) + lego-internal buttons. **Inventory first** (`grep '<button'`); expect a large, area-by-area batched migration — gate per batch, retire the old wrapper last. **Watch for a legacy global element stylesheet** (e.g. a global, unscoped `.btn`/`_btn.scss` with its own color/size variants in a different visual language): **drop those legacy classes** (`class="btn btn-primary"`) *in the same edit* that adds the directive — else the global bleeds into the lego and *ties it on specificity* — and **delete the global stylesheet** once nothing references it. Expect a **mixed look mid-migration** (legacy-styled vs lego) until complete.
- **One component per run.** A slice is run component-by-component, low-blast first (per `migration-plan.md`) — where a "component" is *build-the-lego + migrate-its-consumers*; foundational legos (`esa-icon` + registry) before composites.
- **Mirror the reference faithfully.** Beacon `ui-*` + the hub `.astro` define the API, variants, and structure — don't diverge silently; divergence is a bug.
- **Preserve public API by default.** Consumers shouldn't break unless API migration is the point of the swap.
- **Skill-only production / idempotent.** The swap is produced by this skill; iterate by resetting the working tree, refining the skill, and re-running. Re-running on an already-migrated component is a no-op.
- **Defer the risky forks.** Final brand/theme values are the design spoke's. The hub's Lit web components are a non-Angular *reference*, not an Angular dependency — the app's legos are the permanent Angular implementation, so there's no "consume the published package" step to wait for.
